## Que fue bien

1. **100% del scope entregado.** Las 6 stories se completaron cubriendo la experiencia jugable completa de principio a fin: crear sala, invitar amigos, competir en tiempo real, ver resultados, revancha, reconexion.

2. **Cadena de dependencias limpia.** Stories construidas secuencialmente sin fricciones: datos (2-1) -> infra WebSocket/Redis (2-2) -> motor de juego (2-3) -> UX inmersion (2-4) -> scoring (2-5) -> resiliencia (2-6). Resultado directo de la inversion en arquitectura durante la retro de Epic 1.

3. **Cero regresiones.** 227 tests al final, ninguno roto en el camino. Cada story verifico tests anteriores antes de cerrar.

4. **Patrones reutilizables consolidados.** Lua scripts para atomicidad Redis, Zustand vanilla para estado de alta frecuencia, DOM directo para animaciones, `domain:action` para eventos WebSocket.

5. **Code review atrapo bugs reales.** Race conditions, leaks de Redis, problemas de seguridad JWT — detectados y corregidos durante reviews.

6. **Dev notes excelentes.** Cada story documento decisiones tecnicas, facilitando el contexto para stories posteriores.

7. **Separacion clean de datos.** Redis para estado efimero de juego, PostgreSQL para datos persistentes.

---

## Que no fue bien

1. **Story 2-5 acumulo 12 patches de review sin resolver.** Fue la story con mas deuda tecnica. Analisis posterior revelo que 7 ya estaban resueltos en codigo pero no en tracking.

2. **Tracking de deuda tecnica pobre.** `deferred-work.md` se lleno de items pero nunca se actualizo cuando se resolvian. La retro descubrio que 4 de 29 items eran fantasmas ya resueltos.

3. **Seguridad no priorizada desde el inicio.** Rate limiting y proteccion DDoS no aparecieron hasta la retro, pese a que la hackathon es mundial con historial de ataques.

4. **Mocks fragiles en tests.** Cada story nueva requeria actualizar mocks de stories anteriores. Errores se escondian dentro de try/catch.

5. **Retro de Epic 1 no dejo documento.** Se hizo el trabajo pero no se persistieron las lecciones.

6. **Modelo de agente inconsistente.** Story 2-4 uso Sonnet 4.6 en vez de Opus 4.6 por accidente, no por decision explicita.

---

## Cambios de proceso acordados

1. **Actualizar deferred-work.md en cada story** — si durante una story se arregla un item diferido, marcarlo como resuelto en el mismo PR.

2. **Security NFRs desde la planificacion** — requisitos de seguridad en la story desde el inicio, no como afterthought.

3. **Tests de integracion para el gateway** — deseable pero no bloqueante para Epic 3.

4. **Modelo de agente explicito por story** — documentar que modelo usar. Stories puramente frontend pueden usar Sonnet. Stories que tocan gateway + Redis + shared lib deben usar Opus.

5. **Persistir retrospectivas siempre** — toda retro produce un `.md` en `implementation-artifacts/`.

6. **Fase de validacion funcional post-epica** — levantar la app y probar como usuario real antes de la retro. Flujo: Stories done -> Code reviews done -> Validacion funcional -> Retro -> Prep sprint -> Siguiente epica.

### Limites de rate limiting acordados (ajustados para IP compartida)

**Cloudflare:**
- `/api/auth/*`: 30 req/min por IP
- `/api/*`: 200 req/min por IP
- Security Level: Medium

**@nestjs/throttler:**
- Global: 120 req/min por IP
- `/auth/refresh`, `/auth/code`: 10 req/min por IP
- `/rooms` POST: 5 req/min por IP

**WebSocket guards:**
- Max 3 conexiones WS simultaneas por userId
- Max 2 rooms creados por userId
- CARET_UPDATE: drop si > 25/s por socket

## Decisiones de diseno para Epic 3: Spectator Experience

| Decision | Valor |
|----------|-------|
| Storage de spectators | Hash Redis separado (`room:{code}:spectators`) |
| Broadcast | Separado: players reciben player data, spectators reciben todo |
| Grace period spectators | 0 — liberacion inmediata al desconectarse |
| Tablero en vivo para spectators | Si, ranking reordenandose en vivo, calculo client-side |
| Focus Fade para spectators | No. Ven UI completa + tablero de posiciones |
| Flujo de ingreso | Automatico: < 20 jugadores -> entra como jugador, >= 20 -> entra como espectador. Sin pregunta |
| Cambio de rol | Bidireccional, solo en lobby. Boton "Ser espectador" / "Unirme como jugador" |
| Ingreso a sala llena | Automaticamente como espectador |

### Cambio respecto al epic original

La Story 3-3 original decia: "spectator en post-match results screen clicks to join next match". La decision de la retro cambia esto a: **cambio de rol bidireccional solo en lobby**, sin paso extra de preguntar al entrar. Razon: agregar un paso "jugador o espectador?" al unirse es friccion innecesaria. El sistema decide automaticamente y el usuario puede cambiar en el lobby si quiere.
